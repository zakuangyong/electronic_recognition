# Option C Refined 第二阶段精修设计

**日期**: 2026-06-25
**目标**: 在已完成 `option-c-refined.html` 整页对齐的基础上，继续执行两类改造：`视觉收口` 与 `组件去壳`，并明确以 `识别工作台` 作为主攻页面，同时同步推进 `知识库管理` 与 `图纸检索` 的精修。

## 背景

当前三页已经完成第一阶段改造：

- 页面外壳已对齐到 `topbar / wrap / screen / screen-header / canvas / analysis-shell / toolbar / board / panel`
- 三页已可运行
- 视图测试与类型检查已通过

但当前实现仍存在两类问题：

1. **视觉收口还不够严格**
   - 间距、比例、控件密度、panel 层级虽然已接近 `option-c-refined.html`，但还没有达到“静态稿级”一致
2. **组件边界感仍然可见**
   - 现有 `UploadPanel / TaskStatusCard / ResultTabs / RuleEditor / SearchToolbar` 等 Vue 组件在视觉上仍像被拼接进页面的模块，而不是同一张 screen 中自然生长出的 panel

用户已明确要求：

- 一次执行两类改造
- 两部分都按当前设计执行
- 以 `识别工作台` 为主攻页

## 本轮范围

本轮是对上一份 `Option C Refined 全量对齐设计` 的第二阶段补充，不替代上一轮 spec，而是在其基础上做更严格的页面打磨。

对应上一轮 spec：

- `2026-06-25-option-c-refined-full-alignment-design.md`

本轮新增目标：

- 更严格贴近 `option-c-refined.html`
- 进一步削弱组件拼接感
- 让三页更像完整分析屏，而不是“带静态稿皮肤的业务组件容器”

## 总体策略

采用“**两类改造并行 + 单页主攻**”：

- **视觉收口**
  - 压缩间距
  - 统一卡片层级
  - 收紧标题、标签、按钮、输入框、指标卡尺寸
  - 调整主次区块比例，使版面更像静态稿

- **组件去壳**
  - 削弱组件独立标题、边框、说明文本和默认表单感
  - 让业务组件服从 page-level panel 结构
  - 用户看到的是完整分析屏，而不是一组子组件

- **主攻优先级**
  - 第一优先：`识别工作台`
  - 第二优先：`知识库管理`
  - 第三优先：`图纸检索`

## 视觉基准

唯一视觉真值源仍然是：

- `d:\project\electronic_recognition\.superpowers\brainstorm\session-1782390878\content\option-c-refined.html`

本轮重点对齐的不是外壳命名，而是以下细节：

- toolbar 的高度与信息密度
- metric-grid 的卡片比例和数字权重
- dense-grid 的主次分区
- panel-title 的紧凑度与标签存在感
- result-card 的留白、圆角、字重和 badge 密度
- field / control / chip / meta 的高度与字级

## 页面设计

### 1. 识别工作台

这是本轮主攻页面。

#### 视觉收口目标

- toolbar 更像静态稿中的紧凑工具条，而不是上传组件容器
- `metric-grid` 更接近静态稿中的四枚指标卡：
  - 高度更低
  - 内边距更小
  - 数字权重更集中
- `dense-grid` 更强调主次：
  - 左侧 `图纸预览` 成为绝对主体
  - 右侧 `结果摘要` 缩成辅助分析面板
- 提示文案与状态反馈尽量内聚，不额外占用大块空间

#### 组件去壳目标

- `UploadPanel`
  - 去掉“这是一个独立上传卡”的感知
  - 更像 toolbar 中的输入动作条
- `ResultTabs`
  - 去掉明显的 tab 容器边界感
  - 让它更像主画布的一部分
- `TaskStatusCard`
  - 从“统计卡片组件”进一步收敛成静态稿中的摘要 panel

#### 必须达到的效果

- 用户第一眼看到的是 `图纸预览`
- 结果摘要是辅助区，而不是与主画布争抢主体
- 首屏整体更像“分析屏”，更少后台页痕迹

### 2. 知识库管理

#### 视觉收口目标

- 左栏目录树进一步压缩
  - filter 更短
  - 列表项更紧凑
  - tab 与 action 高度更接近静态稿
- 主区两列 panel 比例更稳定
  - 编辑器区更明确是主体
  - 素材对照区更明确是辅区
- 底部 `试运行结果` 更像独立结果板，而不是附属表单段落

#### 组件去壳目标

- `KnowledgeListPanel`
  - 从“列表组件”转成“目录树面板”
- `ComponentEditor / RuleEditor`
  - 减少表单节块的独立感
  - 更贴近静态稿里的 panel 内编辑内容
- `RuleEditor`
  - 让 `试运行结果` 区更像输出区，而不是配置区延长段

### 3. 图纸检索

#### 视觉收口目标

- 左栏检索条件区收紧为更明确的 `220px` 分析侧栏
- 条件字段更紧凑
- 顶部 metric 更像静态稿中的数据条，而不是功能状态卡
- `结果列表` 的卡片边距、字级、badge 密度进一步贴近静态稿

#### 组件去壳目标

- `SearchToolbar`
  - 弱化表单区独立感
  - 更像检索条件 panel 的内部内容
- `SearchResultList`
  - 更像 screen 主区的结果板
- `SearchResultCard`
  - 更像静态稿的 result-card
  - 降低“可复用组件”的默认感

## 组件边界策略

本轮仍然不改后端协议和数据流。

允许修改的层面：

- page-level 模板
- page-level / component-level CSS
- 展示型组件的标题、边框、布局层级
- 少量组件模板结构，用于去壳

不做的事情：

- 不改 API 协议
- 不新增后端功能
- 不重写 composable
- 不做与当前三页无关的重构

## 结论

第二阶段精修不是重做第一阶段，而是在已经对齐整页骨架的前提下，进一步实现两件事：

- 更严格地贴近 `option-c-refined.html`
- 更彻底地消除组件拼接感

执行策略是“三页并行、工作台主攻”，确保本轮既有全局一致性，也能在最关键页面上体现明显质量提升。
