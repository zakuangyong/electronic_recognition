# 图纸检索结果列表重设计

**日期**: 2026-06-27
**目标**: 将 `图纸检索` 页中的 `结果列表` 从“识别结果汇总卡片”重设计为“检索结果列表”，并以“打开图纸 + 悬浮大预览层”为主交互闭环。

## 背景

当前 `图纸检索` 页虽然已经具备左侧检索条件、右侧结果区和卡片化结果展示，但整体视觉和交互心智仍偏向“分析屏结果汇总”，而不是“检索结果列表”。

当前存在的核心问题：

- 顶部 `metric-grid` 与 `目标区域 / 主结果区 / standard` 等文案过于强调页面壳层
- 结果卡阅读路径偏向“标题 + 大量标签 + 查看结果”
- `查看结果` 的动作语义指向结果页，而不是图纸本身
- 结果卡片更像业务摘要卡，而不是检索输出项
- 命中原因没有被结构化表达，用户需要自行从标签中反推

用户已明确确认以下方向：

- 采用“`搜索结果流` + `轻量快捷动作带`”的混合方案
- 主动作从 `打开结果` 改为 `打开图纸`
- 点击主动作后，不跳转为结果页主流程，而是展示 `悬浮大预览层`
- 结果列表需要显著更像“检索页”，而不是“识别列表汇总”

## 设计结论

本次重设计的唯一推荐方向为：

- **结果区定位**：从“分析主结果区”切换为“查询输出区”
- **结果卡心智**：从“结果摘要卡”切换为“检索命中项”
- **交互闭环**：从“查看结果页”切换为“打开图纸并预览命中页”

具体表达为：

- 保留左侧检索条件栏
- 弱化顶部壳层型指标卡
- 将右侧主区改造成标准纵向检索结果流
- 每条结果强调 `为什么命中`、`命中了哪里`、`如何打开图纸`
- 通过悬浮大预览层承接图纸查看与页码定位

## 视觉目标

### 1. 更强的检索感

用户进入页面后，第一眼应该感知到：

- 我正在看某个查询的结果
- 当前共有多少结果
- 结果是如何排序的
- 每条结果为什么会出现在这里

而不应该首先感知到：

- 这是一个分析屏主结果区
- 这是某种标准模式的业务卡片

### 2. 更弱的汇总感

结果卡不再强调：

- 大量并列 badge
- 模块化标题区
- 业务组件式 panel 壳层

结果卡应该更像：

- 搜索引擎结果
- 文档检索列表
- 以命中证据驱动的文档浏览入口

### 3. 更明确的对象

页面中的“被打开对象”必须从 `结果` 切换成 `图纸`。

因此：

- 主按钮文案为 `打开图纸`
- 辅助动作围绕图纸预览展开
- 结果页仅作为次级深链能力存在，不再作为结果列表的首要动作

## 页面结构调整

### 1. 页面头部

现有 `screen-header` 与结果区顶部保留，但语义调整为“检索结果头”。

推荐结构：

- 查询回显
- 结果总数
- 去重状态
- 排序方式
- 当前可用筛选状态

不再强调以下信息作为首屏主内容：

- `目标区域`
- `主结果区`
- `standard`

这些信息如果仍需保留，只能降级到调试态或隐藏在次级说明中。

### 2. 顶部摘要区

现有 `metric-grid--search` 需要从四张卡片改造成一条轻量结果信息条，视觉上更像搜索页的“查询信息栏”。

建议内容：

- `查询: xxx`
- `找到 N 条结果`
- `已折叠历史版本`
- `排序: 相关度优先`

可选的轻量切换芯片：

- `全部结果`
- `精确命中优先`
- `语义命中优先`
- `含组合规则`
- `仅显示有页码`

### 3. 结果列表区

结果列表保持纵向单列，强调高密度扫描。

每条结果采用“两段式骨架”：

- 左侧主内容区
- 右侧轻量动作带

主内容区负责解释：

- 这是什么图纸
- 为什么命中
- 命中在哪些页
- 命中证据是什么

动作带负责承接：

- 打开图纸
- 定位首个命中页
- 展开更多命中

## 结果卡设计

### 1. 信息层级

每条结果卡按以下顺序展示：

1. 序号
2. 图纸标题
3. 相关度
4. 图号 / 修订 / 项目或系统 / 命中页码
5. 命中摘要片段
6. 命中理由
7. 证据标签
8. 轻量状态信息
9. 快捷动作带

该顺序必须保证用户扫描路径为：

`结果排序 -> 标题 -> 命中理由 -> 打开图纸`

而不是：

`标题 -> 标签堆 -> 业务按钮`

### 2. 标题区

标题区由三部分构成：

- 左侧序号
- 中间标题与元信息
- 右上角 `相关度`

调整要求：

- 分数文案显式写成 `相关度`
- 相关度是排序解释，不是业务状态 badge
- 序号强化“第几条结果”的搜索心智

### 3. 元信息

只保留用户在检索场景下最关心的信息：

- 图号
- 修订版本
- 项目名或系统名
- 命中页码

以下信息降级处理：

- `result_id`
- 冗长的 filename
- 大段系统内标识

这些信息如需保留，只放到卡片底部轻量元信息区，不占据主视觉。

### 4. 摘要片段

摘要区必须优先显示命中上下文，而不是笼统摘要。

要求：

- 尽量展示命中词周围的自然语言上下文
- 如果命中来自规则说明，则展示规则相关片段
- 如果命中来自元件表，则展示元件代号和所在页
- 字数控制在 1 到 3 行，避免变成摘要大段落

### 5. 命中理由

每条结果必须新增一个结构化“命中理由”区块，用一句话解释该结果为何进入当前排序位置。

示例表达：

- 查询词精确命中 `F1 / K11`，同时命中 `电动机启动与保护组合`
- 语义上接近“热继电器保护回路”，并在摘要中出现近义描述
- 命中页码明确，且规则说明文本完整

该区域应使用浅底色区块或左侧强调线，与普通摘要区分开。

### 6. 证据标签

标签从“信息堆叠”改为“证据补充”。

规则：

- 单卡最多展示 3 到 5 个标签
- 标签按价值排序
- 标签类别仅保留：
  - 精确命中
  - 语义/规则命中
  - 命中块类型

不保留的问题：

- 大量重复元件标签并列展示
- 同级信息过多导致用户无法判断主次

### 7. 轻量状态区

卡片底部保留一行轻量状态信息，用于承载次级信息：

- 命中证据数
- 最近更新时间
- 历史版本折叠数
- 是否支持展开全部命中

该区域不做主视觉强化，只作为辅助说明。

## 快捷动作带

### 1. 动作定位

结果卡右侧保留细窄动作带，而不是做成完整三列资产卡。

作用：

- 保留足够强的操作闭环
- 不破坏搜索结果流的纵向阅读感
- 避免页面重新回到“档案卡/汇总卡”模式

### 2. 动作优先级

建议固定为三类动作：

1. `打开图纸`
2. `定位首个命中页`
3. `展开更多命中`

其中：

- `打开图纸` 为主按钮
- `定位首个命中页` 为次按钮
- `展开更多命中` 可为次按钮或卡片内文本操作

### 3. 不再使用的主动作

以下动作不再作为首要 CTA：

- `查看结果`
- `进入结果页`

如果系统仍然需要进入 `ResultView`：

- 仅保留为二级入口
- 不应占据主按钮位

## 悬浮大预览层

### 1. 交互目标

用户点击 `打开图纸` 后，应直接看到图纸内容，而不是跳转到另一个结果容器。

因此采用“悬浮大预览层”作为首选交互形态。

### 2. 预览层结构

预览层建议包含：

- 顶部标题栏
  - 图纸标题
  - 图号
  - 当前页码
  - 关闭按钮
- 主预览区
  - 大尺寸图纸页面预览
- 命中导航区
  - 命中页列表
  - 每个命中的简短原因
  - 一键跳转到该命中页
- 辅助操作区
  - 上一页 / 下一页
  - 放大 / 缩小
  - 打开完整结果页（次级）

### 3. 打开逻辑

打开预览层时：

- 默认定位到首个最强命中页
- 如果存在多个命中页，展示页码导航
- 如果命中页未知，则默认显示图纸首页并提示“未定位具体页码”

### 4. 与结果列表的关系

预览层是列表的上层交互，不是页面的新主布局。

因此不采用：

- 常驻右栏预览
- 页面上下固定分屏
- 直接跳独立预览页作为主路径

这样可以保留：

- 用户仍然知道自己在当前检索列表里
- 关闭后可继续浏览其它结果

## 文案策略

### 1. 强化检索语言

建议主文案优先使用：

- `查询`
- `结果`
- `相关度`
- `命中理由`
- `命中页`
- `打开图纸`

### 2. 弱化分析壳语言

弱化或移除：

- `目标区域`
- `主结果区`
- `standard`
- `查看结果`

### 3. 按钮文案

推荐按钮文案：

- `打开图纸`
- `定位命中页`
- `展开命中`
- `查看完整结果`（仅次级）

## 组件与文件影响范围

本次设计不改后端协议，但会影响前端的页面模板和展示组件。

核心影响点：

- `web/src/views/SearchView.vue`
  - 结果页头和主区语义重组
- `web/src/components/search/SearchResultList.vue`
  - 顶部摘要区改为结果信息条
- `web/src/components/search/SearchResultCard.vue`
  - 卡片结构改为搜索结果流
  - 主按钮改为 `打开图纸`
  - 新增命中理由区和动作带
- `web/src/app/styles/search.css`
  - 重写结果列表与预览层相关样式

可能新增：

- 图纸预览悬浮层组件
- 命中页导航子组件

## 数据映射要求

前端优先复用现有字段：

- `score`
- `matched_pages`
- `snippet`
- `match_sources`
- `matched_chunk_types`
- `matched_components`
- `matched_combinations`
- `preview_url`

前端需要做的不是增加数据量，而是重新组织展示层级。

具体策略：

- `score` 映射为 `相关度`
- `matched_pages` 映射为命中页码与定位动作
- `snippet` 映射为摘要片段
- `match_sources + matched_chunk_types + 局部命中项` 组合成命中理由与证据标签
- `preview_url` 不再直接表达为“查看结果”的主入口，而作为图纸打开链路的一部分

## 不做的事情

本次设计明确不包括：

- 修改后端检索打分逻辑
- 修改 API 协议
- 重做左侧检索字段能力
- 引入批量收藏、导出、对比等资产管理功能
- 把页面改造成永久双栏预览工作台

## 验证标准

设计落地后，用户应能明显感受到以下变化：

1. 首屏更像检索页，而不是分析屏
2. 每条结果更像搜索命中项，而不是识别摘要卡
3. 用户能快速理解“为什么命中”
4. 主动作明确变成“打开图纸”
5. 图纸预览通过悬浮大预览层完成，不打断列表浏览

## 验收标准

视为完成，需要同时满足：

- 页面首屏存在明确查询结果头，而不是壳层型指标卡
- 结果卡主路径为 `标题 -> 命中理由 -> 打开图纸`
- `查看结果` 不再是主按钮
- 悬浮大预览层可承接图纸查看与命中页定位
- 结果列表整体观感明显弱化“识别汇总感”
- 页面更接近“图纸检索结果页”的用户心智

## 结论

本次重设计不是简单压缩卡片样式，而是一次结果列表心智切换：

- 从“分析结果板”切换为“检索结果流”
- 从“查看结果”切换为“打开图纸”
- 从“卡片摘要堆叠”切换为“命中理由驱动”
- 从“页面内业务组件”切换为“查询输出界面”

最终推荐方案为：

- **搜索结果流骨架**
- **右侧轻量快捷动作带**
- **悬浮大图纸预览层**

这是当前最能同时满足“检索感增强”“图纸导向交互”“不回退成汇总卡片”的方案。
