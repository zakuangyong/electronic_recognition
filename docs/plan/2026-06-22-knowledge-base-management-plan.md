# 知识库管理与组合元件编辑实施计划

## 1. 目标与范围

在现有电气元件识别工作台中补齐知识库的可视化维护能力：

1. 在页面右上角新增“知识库管理”入口。
2. 提供单元件知识库的查询、新增、编辑、启停/删除和图片维护能力。
3. 提供组合元件知识库的查询、新增、编辑、启停和删除能力。
4. 提供组合元件编辑器，允许用户从单元件知识库中人工选择成员、定义角色和数量，形成可被识别流程使用的组合元件规则。
5. 所有写操作通过后端 API 完成，禁止前端直接拼接或覆盖 JSON 文件。
6. 保持现有识别接口和已有知识库数据向后兼容。

本计划将“手动已形成组合元件”解释为：用户人工创建或修改一个组合元件定义/匹配规则，保存后参与后续识别。首期不包含对某一次历史识别结果中 `detected_combinations` 的人工改写；如需要，可在后续增加“识别结果纠正并沉淀为知识”的闭环。

## 2. 现状梳理

### 2.1 前端

- 主页面位于 `src/electronic_recognition/static/index.html`。
- 交互逻辑位于 `src/electronic_recognition/static/app.js`。
- 样式位于 `src/electronic_recognition/static/styles.css`。
- 页头右侧当前只有服务状态，适合增加“知识库管理”入口。
- 左侧已有只读的“知识库预览”，支持单元件/组合元件切换、搜索和分页，但不支持写操作。
- 后端已经注册 `/knowledge` 页面路由，但 `src/electronic_recognition/static/knowledge.html` 尚不存在，当前访问会失败。

### 2.2 后端与数据

- 单元件模型为 `ComponentSample`，定义在 `src/electronic_recognition/models.py`。
- 单元件存储在 `data/index/components.json`，图片存储在 `data/index/assets/components/`。
- 组合元件规则模型为 `CustomRule`、`RuleMember`，定义在 `src/electronic_recognition/custom_rules.py`。
- 组合规则存储在 `data/index/custom_rules.json`，图片存储在 `data/index/assets/rules/`。
- 当前已有 157 个单元件、3 个内置组合规则和 1 个声明式自定义组合规则。
- 当前 API 只有知识库列表和图片读取接口，没有新增、修改、删除、上传、校验接口。
- 声明式组合规则目前通过 `code_patterns` 和 `label_keywords` 匹配，尚不能稳定表达“明确选择知识库中的某几个单元件”。

## 3. 产品与交互方案

### 3.1 页头入口

调整 `index.html` 页头右侧结构：

- 将服务状态与“知识库管理”按钮放入统一的操作区。
- 按钮使用图标加文字，不使用仅图标入口。
- 点击后跳转到 `/knowledge`。
- 小屏宽度下保留文字；极窄屏可换行，但不能遮挡服务状态。
- 按钮具备 hover、active、focus-visible 状态，最小可点击高度 40px。

推荐结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ Electronic Recognition                 [知识库管理] ● 服务在线 │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 知识库管理页

使用独立页面而不是弹窗。该功能包含列表、筛选、图片和复杂编辑表单，独立页面更适合保持清晰的信息层级，也便于后续扩展批量导入与版本管理。

页面整体结构：

```text
页头：返回识别工作台 / 页面标题 / 数据状态

统计区：单元件总数 / 组合元件总数 / 已启用组合 / 数据更新时间

主标签：
  1. 单元件
     - 搜索、类型筛选、来源筛选
     - 列表/卡片
     - 新增单元件
     - 查看、编辑、删除

  2. 组合元件
     - 搜索、引擎筛选、状态筛选、作用域筛选
     - 列表
     - 新建组合元件
     - 查看、编辑、复制、启停、删除
```

首期采用桌面端“左侧列表 + 右侧详情/编辑抽屉”或“列表 + 全宽编辑区”模式。宽度小于 900px 时改为单列，编辑器作为全屏内容区展示，避免横向挤压。

### 3.3 单元件管理

列表展示：

- 预览图
- 名称
- ID
- 元件类型
- 型号
- 别名数量
- 变体图数量
- 来源
- 操作

支持的操作：

- 新增：填写元数据并上传主图。
- 编辑：修改名称、类型、型号、定义、标准、别名、备注、来源。
- 图片维护：替换主图，增加或移除变体图。
- 复制：复制元数据生成新 ID，图片按需复用或重新上传。
- 删除：二次确认；内置/标准数据可根据数据来源配置为只读或只允许停用。

表单校验：

- `id` 必填，只允许字母、数字、下划线和连字符，创建后默认不可修改。
- `label`、主图必填。
- ID 必须唯一。
- 图片仅允许 PNG/JPEG/WebP，限制单文件大小，服务端校验真实 MIME 类型。
- 别名和标准做去空、去重处理。
- 删除前检查该单元件是否被组合规则引用；被引用时默认阻止删除，并展示引用规则。

### 3.4 组合元件管理

列表展示：

- 预览图
- 名称与 ID
- 内置/自定义标识
- 成员数量
- 作用域
- 置信度
- 启用状态
- 操作

规则权限：

- `engine=builtin`：首期只允许查看详情和启停，不允许修改成员判定逻辑或删除。
- `engine=declarative`：允许完整新增、编辑、复制、启停和删除。

### 3.5 组合元件编辑器

编辑器分为四个区域，采用渐进披露，避免一次展示过多技术字段。

#### A. 基本信息

- 组合 ID
- 名称
- 描述
- 别名
- 备注
- 来源
- 预览图
- 是否启用
- 默认置信度

#### B. 匹配范围

- `same_page`：所有成员必须出现在同一页。
- `document`：成员可分布在同一文档不同页。

默认选择 `same_page`，因为误匹配风险更低。

#### C. 成员编排

用户通过“添加成员”从单元件知识库中搜索并选择元件，然后配置：

- 角色名称，例如“短路保护”“接触器控制”。
- 选中的单元件，可多选。
- 最小数量。
- 可选的代号正则。
- 可选的名称关键词。

每个成员行支持：

- 上移/下移调整展示顺序。
- 复制成员。
- 删除成员。
- 展开高级匹配条件。

首期不做自由画布连线。组合规则的核心是成员集合与匹配约束，自由布局并不参与现有识别判定，过早引入画布会显著增加复杂度。预览图可由用户上传；后续如需要表达拓扑关系，再增加锚点和连线模型。

#### D. 校验与预览

保存前显示：

- 规则摘要。
- 成员数量和最小实例总数。
- 是否存在空角色、重复角色、无任何匹配条件的成员。
- 被引用单元件是否仍存在。
- 正则表达式是否合法。

提供“试运行”入口：选择一个已有识别结果或提交当前规则到校验接口，返回匹配到的组合、成员和证据，但不保存规则。

## 4. 数据模型调整

### 4.1 单元件

沿用 `ComponentSample`，首期不改变已有 JSON 主体格式。建议补充管理层状态字段：

```python
enabled: bool = True
created_at: str = ""
updated_at: str = ""
```

兼容策略：

- 旧记录缺少 `enabled` 时按 `True` 处理。
- 时间字段缺失时允许为空。
- 识别候选和名称修正阶段过滤 `enabled=False` 的记录。

如首期不需要单元件启停，可暂不增加 `enabled`，只实现受保护删除。

### 4.2 组合规则成员

扩展 `RuleMember`：

```python
@dataclass(slots=True)
class RuleMember:
    role: str
    min_quantity: int = 1
    component_ids: list[str] = field(default_factory=list)
    code_patterns: list[str] = field(default_factory=list)
    label_keywords: list[str] = field(default_factory=list)
```

匹配语义：

- 同一成员内部的 `component_ids`、`code_patterns`、`label_keywords` 采用 OR。
- 不同成员之间采用 AND。
- `component_ids` 优先根据识别结果的 `reference_id` 精确匹配。
- 至少配置一种匹配条件，否则规则校验失败。
- 老规则没有 `component_ids` 时保持原有行为。

这样可以将“人工从知识库选择成员”的操作稳定映射为机器可执行规则，而不是由前端猜测正则表达式。

### 4.3 数据版本

- `components.json` 版本由当前版本升级一位，仅在实际增加字段时升级。
- `custom_rules.json` 版本由 1 升级为 2。
- 读取器对缺失版本和旧版本保持兼容。
- 提供一次性迁移脚本或在服务启动时只读兼容，避免自动静默改写用户数据。

## 5. 后端设计

### 5.1 分层

新增知识库写服务，避免把文件操作继续堆积在 `api.py`：

```text
src/electronic_recognition/
  knowledge_store.py          # 单元件 CRUD、校验、原子写入
  custom_rule_store.py        # 组合规则 CRUD、引用检查、原子写入
  knowledge_schemas.py        # FastAPI/Pydantic 请求响应模型
  api.py                      # 路由和 HTTP 错误映射
```

如果希望减少文件数量，也至少应把“加载—校验—锁定—原子替换”抽成独立存储服务。

### 5.2 API

#### 单元件

```text
GET    /api/knowledge
GET    /api/knowledge/{component_id}
POST   /api/knowledge
PUT    /api/knowledge/{component_id}
DELETE /api/knowledge/{component_id}
POST   /api/knowledge/{component_id}/images
DELETE /api/knowledge/{component_id}/images/{image_id}
GET    /api/knowledge/{component_id}/image
GET    /api/knowledge/{component_id}/images/{image_id}
```

列表接口增加可选参数：

```text
q, component_type, source, page, page_size
```

为兼容现有前端，未传分页参数时可继续返回完整 `items`；管理页必须使用服务端分页，避免知识库增长后一次加载所有图片。

#### 组合规则

```text
GET    /api/custom-rules
GET    /api/custom-rules/{rule_id}
POST   /api/custom-rules
PUT    /api/custom-rules/{rule_id}
PATCH  /api/custom-rules/{rule_id}/enabled
DELETE /api/custom-rules/{rule_id}
POST   /api/custom-rules/{rule_id}/image
POST   /api/custom-rules/validate
POST   /api/custom-rules/test
```

约束：

- 创建接口只允许 `engine=declarative`。
- 内置规则拒绝 PUT 和 DELETE。
- 删除单元件前返回其组合规则引用列表。
- 更新或删除后，`/api/config` 中计数立即反映最新数据。

### 5.3 文件写入安全

JSON 文件是当前系统的事实数据源，写操作必须做到：

1. 使用进程内锁串行化同一知识库的写操作。
2. 先完整加载并校验现有 JSON。
3. 在内存中应用变更。
4. 写入同目录临时文件。
5. `flush`/`fsync` 后使用原子替换覆盖目标文件。
6. 修改前生成带时间戳的备份，保留最近若干份。
7. 图片先写临时文件，校验成功后移动到受控资产目录。
8. API 只接受文件内容，不接受用户提交的任意服务器路径。
9. 删除时只允许删除知识库根目录下、且确属当前记录引用的资产。

需要明确当前部署仅适合单进程文件存储。若未来启用多 worker 或多实例，应迁移到数据库/对象存储，或增加跨进程文件锁。

### 5.4 错误响应

统一返回：

```json
{
  "detail": {
    "code": "component_referenced",
    "message": "该单元件被 2 个组合规则引用，无法删除。",
    "fields": {},
    "references": ["rule-a", "rule-b"]
  }
}
```

建议错误码：

- `invalid_payload`
- `duplicate_id`
- `not_found`
- `builtin_rule_readonly`
- `component_referenced`
- `invalid_regex`
- `invalid_image`
- `write_conflict`
- `storage_error`

## 8. 测试计划

### 8.1 单元测试

新增建议：

```text
tests/test_knowledge_store.py
tests/test_custom_rule_store.py
tests/test_knowledge_api.py
```

覆盖：

- 创建、修改、删除单元件。
- 重复 ID、非法 ID、非法图片。
- 变体图增加和移除。
- 被组合规则引用的单元件不可直接删除。
- 创建、修改、复制、启停、删除声明式组合规则。
- 内置规则不可修改或删除。
- 非法正则被拒绝。
- JSON 原子写入失败时原文件不损坏。
- 旧版本数据可以加载。
- `component_ids` 能按 `reference_id` 命中。
- 多条件 OR、成员间 AND、`min_quantity`、`same_page`、`document` 语义正确。

### 8.2 API 集成测试

- multipart 图片上传。
- CRUD 状态码和统一错误体。
- 列表筛选、分页和排序。
- 删除引用冲突返回 409。
- 并发保存时不会产生截断 JSON。
- 保存后 `/api/config` 计数更新。
- 试运行不写入规则文件。

### 8.3 前端验收

- 1440px、1024px、768px、375px 宽度。
- 键盘可以完成标签切换、搜索、列表选择、编辑和保存。
- 所有图标按钮有可访问名称。
- focus 状态清晰，文本对比度不低于 WCAG AA。

- 长名称、长 ID、多个别名不会破坏布局。
- 未保存修改不会因误操作丢失。

### 8.4 端到端业务用例

1. 新增一个单元件并上传主图。
2. 用该单元件和已有单元件创建一个自定义组合。
3. 通过试运行验证规则匹配。
4. 保存规则并执行一次识别。
5. 在组合识别结果中看到新规则产生的组合。
6. 禁用该规则后再次识别，确认不再输出该组合。
7. 尝试删除被规则引用的单元件，确认系统阻止并给出引用来源。

## 9. 验收标准

- 主页面右上角存在清晰、可键盘访问的“知识库管理”入口。
- `/knowledge` 可正常打开，不再引用缺失文件。
- 单元件可以查询、新增、编辑、维护图片和受保护删除。
- 声明式组合元件可以查询、新增、编辑、复制、启停和删除。
- 用户可以从现有单元件中选择成员并形成组合规则，无需手写 JSON。
- 新规则保存后可参与识别，禁用后不参与识别。
- 内置规则不会被误删或修改核心逻辑。
- 任意失败写操作不会破坏原 JSON 或遗留越权路径文件。
- 现有识别流程、只读知识库预览和已有组合规则测试保持通过。
- 管理页具备加载、空、错误、保存成功、删除确认和未保存提醒状态。

## 10. 风险与后续演进

### 主要风险

- 文件型存储不适合多实例并发写入。
- 单元件被规则引用时，修改 ID 或删除会造成规则失效。
- 规则条件过宽会显著增加组合误报。
- 用户上传图片可能带来大文件、错误格式和路径安全问题。
- 157 个单元件尚可前端浏览，但继续增长后必须坚持服务端分页和图片懒加载。

### 缓解措施

- 单进程锁、原子写入、备份和引用完整性检查。
- ID 创建后不可修改；如必须修改，提供显式“迁移引用”操作。
- 保存规则前执行静态校验，并通过试运行展示命中证据。
- 服务端解码并重新编码图片，生成受控文件名。
- 默认 `same_page`，高级用户才选择 `document`。

### 后续能力

- 批量导入/导出 CSV、JSON、ZIP。
- 知识库变更历史、审计日志和一键回滚。
- 从识别结果选中多个实例，一键沉淀为组合规则。
- 组合拓扑、成员相对位置和连线约束。
- 人工纠正识别结果并形成训练/评测样本。
- 数据量或并发提升后迁移到 SQLite/PostgreSQL + 对象存储。

## 11. 推荐实施顺序与工作量

推荐按“存储安全 → API → 浏览页 → 单元件编辑 → 组合编辑 → 识别回归”推进，避免先做出可编辑 UI，却让前端直接覆盖 JSON。

粗略工作量：

| 模块 | 预计工作量 |
|---|---:|
| 数据模型、存储服务、迁移兼容 | 1.5～2 人日 |
| CRUD、上传、校验、试运行 API | 1.5～2 人日 |
| 管理页框架与响应式列表 | 1～1.5 人日 |
| 单元件编辑与图片维护 | 1～1.5 人日 |
| 组合元件编辑器 | 2～3 人日 |
| 测试、回归、文档与缺陷修复 | 1.5～2 人日 |
| 合计 | 8.5～12 人日 |

若首期只做“查看 + 新增/编辑声明式组合规则”，暂不做单元件删除、变体图维护和规则试运行，可压缩到约 4～6 人日。
